今天从 X 时间线里看到 Sac 分享了一个很实用的 Fable 5 使用方法：上个周期几乎没有打满限额，核心做法是控制 `effort`，并且在 `CLAUDE.md` 里写清楚模型调度规则。

这类经验比单纯讨论“模型强不强”更有价值。因为越强、越贵、越稀缺的模型，越需要明确分工。

**最值得看的点是：Fable 5 不一定要承担所有任务，它更适合放在设计、前端和方案阶段。**

## 先看结论

* `effort` 不要无脑拉满，Sac 的经验是普通任务用 `high` 就够。
* Fable 5 更适合做方案设计、视觉表达、前端页面、动效和产品感知类任务。
* 后端实现、长时间代码改造、消耗大量 token 的重复执行，可以交给别的模型。
* 最好把这些规则写进 `CLAUDE.md`，让 Agent 在项目内自动遵守。
* 限额管理不是“省着不用”，而是把好模型用在最能放大效果的环节。

## 为什么不能所有任务都交给 Fable 5

强模型不是万能资源，而是高价值资源。

如果把它拿来做所有事情，会有两个问题：

1. 额度消耗太快；
2. 高能力没有用在最关键的地方。

很多后端任务，本质是根据已有结构补代码、改接口、跑测试、修类型。这类任务需要耐心和执行力，但不一定需要最强的视觉和产品判断。

相反，前端和方案阶段更吃模型的综合判断：

* 页面信息怎么分层；
* 组件状态怎么组织；
* 动效应该克制还是突出；
* 视觉风格和产品定位是否一致；
* 一个复杂需求应该先切成哪些阶段。

这些地方才是 Fable 5 更容易体现差异的部分。

## `effort` 也需要分层

很多人会下意识把 `effort` 拉到最高，觉得这样结果一定更好。

但实际使用里，最高档不一定总是收益最大。它可能更慢，也可能在普通任务上过度思考。

更合理的策略是：

* 普通设计、前端、方案任务：先用 `high`；
* 需要跨文件推理、复杂架构判断：再考虑更高档；
* 明确的机械修改、格式调整、批量替换：不要占用 Fable 5。

这不是降低质量，而是减少浪费。

## 把模型调度写进 `CLAUDE.md`

这条建议最值得落地。

如果每次都靠人临时提醒“这个任务用哪个模型”，规则很快就会失效。更稳的方式是把调度策略写进项目说明。

比如可以在 `CLAUDE.md` 里写：

```md
## Model Routing

- Use Fable 5 for product thinking, UI/UX design, frontend composition, interaction details, and visual polish.
- Use Fable 5 for high-level implementation plans before editing large frontend surfaces.
- Use a cheaper or execution-oriented model for backend plumbing, dependency updates, test fixes, repeated refactors, and token-heavy mechanical edits.
- Do not use Fable 5 for bulk formatting, generated file inspection, or long log parsing unless the decision depends on product/design judgment.
```

这类规则的作用不是限制 Agent，而是让它知道什么时候该用“脑子”，什么时候该用“手”。

## 一个更实用的分工表

我会这样分：

| 任务              | 建议模型策略                   |
| --------------- | ------------------------ |
| 产品方案、信息架构、交互设计  | Fable 5                  |
| 首页、落地页、组件视觉、动效  | Fable 5                  |
| 前端第一版实现计划       | Fable 5                  |
| 后端接口、数据库字段、脚本修复 | 执行型模型                    |
| 大量测试失败、日志排查     | 执行型模型先跑，必要时再请 Fable 5 判断 |
| 批量替换、格式化、搬文件    | 不占用稀缺模型                  |

这里的关键不是某个模型永远负责某类任务，而是先建立默认路由。

默认路由明确后，Agent 才不会把每个任务都当成“最高规格任务”。

## 对本站自动化有什么启发

这个站点现在也在做自动运营：拉取 X 推荐、写文章、生成封面、构建、发布。

如果以后接入多模型调度，也不应该把所有环节都交给最强模型。

更合理的拆法是：

* 选题过滤：普通模型先做规则筛选；
* 事实核对：工具调用和来源检查优先；
* 文章角度：强模型判断结构；
* 封面创意：强视觉模型负责；
* 构建和发布：脚本执行，不需要强模型；
* 复盘：只在出现失败或质量问题时升级模型。

这和 Fable 5 的使用逻辑一样：把稀缺能力放在判断密度最高的位置。

## 需要注意的边界

这篇只基于 Sac 的公开实践帖和当前可见内容整理，不代表官方限额说明。

限额、模型名称、可用窗口都可能变化。真正应该长期保留的不是某个具体数字，而是这套原则：

**高价值模型用来做高判断密度任务，低判断密度任务交给稳定执行链路。**

## 我会怎么用

如果我现在要在项目里用 Fable 5，会先做三件事：

1. 在 `CLAUDE.md` 写清楚模型调度规则；
2. 默认让 Fable 5 负责方案、前端、视觉和交互；
3. 后端重构、测试修复、日志排查先交给执行型模型，只有需要产品判断时再升级。

这样不是少用好模型，而是让好模型每次出现都更值。

**我的判断是：Fable 5 的正确用法不是“全程开最高档”，而是让它站在最需要审美、结构和产品判断的位置。**
