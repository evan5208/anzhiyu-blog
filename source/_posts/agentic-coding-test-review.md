今天从时间线看到一条关于 AI 代码审阅的讨论，问题很直接：**AI 生成的代码，到底要不要人工审阅？**

这个问题很容易吵成“信 AI”还是“不信 AI”。但更实用的切法其实是：哪些东西应该交给测试，哪些东西才需要人来判断。

**我的结论是：agentic coding 时代，测试应该成为第一审稿人。**

## 先看结论

* 能被自动验证的行为，应该尽量进入自动化测试。
* AI 写单元测试很容易，但复杂集成测试仍然需要人搭建测试环境和关键路径。
* 人工审阅不应该重复做机器能稳定做的检查，而应该关注产品意图、边界条件、架构取舍和风险判断。
* 对 AI Agent 来说，测试不是交付后的补丁，而是持续行动时的方向盘。
* 如果一个需求无法被任何测试、脚本或验收步骤验证，那它就很难交给 Agent 稳定推进。

## 为什么测试比审阅更靠前

传统开发里，code review 往往是提交后的质量关口。

但 AI 生成代码以后，问题变了。Agent 可以很快写出大量改动，也可以很快补测试、改实现、重跑命令。这个速度优势只有在反馈足够明确时才成立。

人工审阅给的是判断，自动化测试给的是边界。

如果边界清楚，Agent 可以自己循环：

1. 修改代码；
2. 运行测试；
3. 读取失败；
4. 修复失败；
5. 再次验证。

这条闭环一旦建立，人的精力就不用花在“这个按钮还能不能点”“这个函数有没有返回空值”这种可重复问题上。

## 人仍然要做什么

测试不是为了取消人工审阅，而是为了让人工审阅站到更高的位置。

人更应该看这些问题：

* 需求是不是被误解了；
* 流程是不是符合真实用户的使用方式；
* 测试覆盖的路径是不是太窄；
* 新抽象是不是过度设计；
* 安全、隐私、成本、性能有没有被低估；
* 这次改动会不会让以后更难维护。

这些问题不能简单交给“npm test 通过”来证明。

但反过来，如果一个 PR 连基本行为都没有自动验证，就直接进入人工审阅，人会被大量低级不确定性拖住。

## 集成测试是关键缺口

AI 现在写单元测试通常不难。真正难的是更贴近真实系统的集成测试。

比如一个博客自动发布流程，单元测试只能证明某个函数处理 Markdown 没问题；但真正要验证的是：

* frontmatter 能被 Hexo 正确解析；
* 封面路径能生成到 `public/img`；
* 首页能展示新文章；
* 搜索索引能命中文章标题；
* GitHub Pages 部署后线上路径返回 200。

这些检查往往需要人先把流程搭出来。搭好以后，Agent 才能反复执行。

这也是为什么复杂集成测试一开始很费事，但长期看很划算：它把人的经验固化成可重复的验收步骤。

## 怎么放进自己的 Agent 工作流

我会把 AI 编程任务拆成三层验证。

第一层是快速本地检查。

比如 lint、typecheck、单元测试、构建命令。它们速度快，适合每次小改后立即跑。

第二层是行为级验证。

比如 Playwright 页面检查、接口回归、关键用户路径、生成文件存在性、资源路径检查。这层更贴近真实结果。

第三层是人工审阅。

人只审那些自动化无法稳定判断的问题：需求是否正确、表达是否合适、风格是否统一、风险是否可接受。

这三层顺序不能倒。先让机器过滤确定性问题，再让人处理判断性问题。

## 对内容站也一样

这个站点的自动化运营其实也是同一个逻辑。

一篇文章不是写完就算完成，而是要验证：

1. 来源没有重复；
2. 内容没有明显敏感或低质风险；
3. Markdown frontmatter 正确；
4. 封面图片存在且路径可访问；
5. `npm run clean && npm run build` 通过；
6. 首页、文章页、搜索索引都能命中；
7. GitHub Pages 线上访问正常。

这些步骤能自动化，就不要只靠肉眼扫一遍。

**Agent 写得越快，验证链路越要硬。**
